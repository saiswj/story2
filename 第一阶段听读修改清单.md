   - 问题：合订版存在 Markdown 标题、文件标题、章节标题重复风险。
   - 修法：导出版统一为“第001章 标题”，删除 `#`、重复章名和多余卷标题。

## 系统提示方括号处理规则

1. 短提示改成正文声源。
   - 原写法：`【听劝系统绑定完成。】`
   - 建议：`脑海里响起一声轻响。系统提示：听劝系统绑定完成。`

2. 重要任务保留系统感，但不机械照读。
   - 原写法：
     - `【任务：不要给许晴送早餐】`
     - `【奖励：记忆力提升】`
     - `【评级：B】`
   - 建议：`系统面板在眼前展开，只列了三行。任务是不要给许晴送早餐，奖励是记忆力提升，评级暂定为B。`

3. 多行面板压缩成2-3句。
   - 任务名、奖励、评级可以保留。
   - 重复状态、说明性文字尽量转述。
   - 不逐条照搬超过3行的系统内容。

4. 后期系统弱化时用旁白化处理。
   - 示例：`系统没有给新任务，只把完成度往上推了一格。`
   - 示例：`提示停在原地，没有替他做选择。`

5. 只有特别短、特别有冲击力的系统提示，可以保留为独立短句。
   - 仍建议去掉方括号。
   - 例：`系统提示：建议采纳。`

核心原则：保留系统爽点和仪式感，但让听众听起来像故事里的声音，而不是截图朗读。

## P1 听读化规则

1. 论坛、评论、弹幕每章超过3条时，改成“旁白概述 + 保留1-2条关键原文”。
2. 系统面板不逐条机械读，按“系统提示/系统面板/提示停住”的口播方式处理。
3. 数据屏、后台、公告只保留最关键三项，其余转述。
4. 条款句超过90字或含多个分号时，拆成短句，并插入人物动作或反应。
5. 多人对白每两三轮补一次说话人或动作锚点。
6. 每章结尾避免“清单、复核、议程、记录”式工作钩，改成下一章外部事件、人物冲突或不可逆动作。
